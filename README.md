# Ховд-457 — Иргэний үйлчилгээний портал

Ховд аймаг дахь Шүүхийн шийдвэр гүйцэтгэх газрын иргэдэд зориулсан цаг захиалга, санал хүсэлт, мэдээллийн систем.

## Бүтэц

```
khovd457/
├── backend/          Node.js + Express API сервер, SQLite өгөгдлийн сан
│   ├── src/
│   │   ├── server.js       Серверийн үндсэн файл
│   │   ├── db/              Схем, seed script
│   │   ├── routes/          auth, queue, feedback, news, admin
│   │   └── util/            JWT, validation
│   ├── package.json
│   └── .env.example
└── frontend/         Цэвэр HTML/CSS/JS (framework шаардлагагүй)
    ├── index.html
    ├── styles.css
    ├── app.js
    └── assets/emblem.svg   ← ЭНД ЖИНХЭНЭ СҮЛДЭЭ ОРЛУУЛНА
```

Backend нь мөн frontend-ийг статик файлаар үйлчилдэг тул **нэг сервер л хэрэгтэй** — тусад нь frontend hosting шаардлагагүй.

## Технологи, яагаад ингэж сонгосон

- **Express** — API сервер.
- **SQLite (node:sqlite, Node.js суурь модуль)** — гадаад package суулгах, native compile хийх шаардлагагүй, файл дээр суурилсан хөнгөн database. 457-ийн хэмжээний байгууллагад (өдөрт хэдэн зуун хүсэлт) энэ хангалттай. Хэрэв ирээдүйд ачаалал их болбол `src/db/index.js`-г PostgreSQL руу шилжүүлэхэд routes-ийн кодыг бараг өөрчлөх шаардлагагүй (SQL query-үүд төстэй).
- **JWT (jsonwebtoken)** — иргэн болон admin-ийн нэвтрэлтийн токен.
- **bcryptjs** — admin нууц үгийг hash хэлбэрээр хадгална (тодорхой текстээр хэзээ ч хадгалагдахгүй).

## Ажиллуулах (локал компьютер дээр)

Шаардлага: **Node.js 22.5+** (`node:sqlite` модуль ашигладаг тул).

```bash
cd backend
npm install
cp .env.example .env
# .env файл дотор JWT_SECRET-ийг санамсаргүй урт мөрөөр солино:
# openssl rand -hex 32

node src/db/seedAdmin.js admin "ХүчтэйНууцҮг123!" "Таны нэр"

npm start
```

Дараа нь `http://localhost:3000` хаягаар нээнэ. Frontend, backend хоёулаа энд ажиллана.

## Жинхэнэ серверт байршуулах (production)

1. **Сервер бэлдэх**: Ubuntu VPS эсвэл байгууллагын дотоод сервер дээр Node.js 22+ суулгана.
2. **Кодоо хуулах**: `git clone` эсвэл `scp`-ээр backend, frontend фолдеруудыг байршуулна.
3. **`.env` тохируулах**: жинхэнэ `JWT_SECRET`-ийг production дээр л үүсгэж тавина (repo-д хэзээ ч бүү commit хийгээрэй).
4. **Admin бүртгэх**: `node src/db/seedAdmin.js <username> <password>` — анхны нэвтрэх мэдээллийг зөвхөн итгэмжит хүн мэднэ.
5. **Процесс удирдлага**: `pm2` эсвэл `systemd`-ээр серверийг байнга ажиллуулна:
   ```bash
   npm install -g pm2
   pm2 start src/server.js --name khovd457
   pm2 save
   pm2 startup
   ```
6. **HTTPS / reverse proxy**: nginx-ээр урд талд нь HTTPS (Let's Encrypt) тавьж, `proxy_pass http://127.0.0.1:3000;` гэж чиглүүлнэ. Иргэдийн регистрийн дугаар, утасны дугаар дамждаг тул **HTTPS заавал шаардлагатай**.
7. **Нөөцлөлт (backup)**: `backend/data/khovd457.db` файлыг өдөр бүр нөөцөл (cron + `cp` эсвэл rsync). Энэ файл дотор бүх иргэдийн мэдээлэл байгаа.

## Аюулгүй байдлын анхаарах зүйлс — production-д заавал засах

Энэ бол бодит ашиглалтад бэлэн эхлэл (MVP) — гэхдээ дараах зүйлсийг байгууллагын мэдээллийн аюулгүй байдлын шаардлагад нийцүүлж сайжруулахыг зөвлөж байна:

- **Иргэний баталгаажуулалт**: одоогийн систем регистр+утасны дугаараар л таньдаг бөгөөд энэ нь хуурамч мэдээлэл оруулахаас бүрэн хамгаалахгүй. Бодит SMS OTP баталгаажуулалт эсвэл **e-Mongolia / Харилцан итгэлцлийн систем**-тэй холбох нь илүү найдвартай.
- **Rate limiting**: одоо энгийн IP-суурьтай хязгаарлалт л бий (`express-rate-limit`). Олон нийтэд нээлттэй болгохоос өмнө илүү нарийвчилсан хамгаалалт (жишээ нь Cloudflare, fail2ban) нэмэхийг зөвлөж байна.
- **Admin эрх**: одоо ганц admin хэрэглэгч байх боломжтой систем. Хэрэв олон ажилтан admin эрхтэй байх шаардлагатай бол `admins` хүснэгтэд эрхийн түвшин (role) нэмж болно.
- **Log хяналт**: production-д алдааны log-ийг зөвхөн console-д бус, файл эсвэл log service-д бичихийг зөвлөж байна.

## Онцлог логик — Урт хугцааны эргэлт (72 цаг, 3 хүн)

`backend/src/routes/queue.js`-ийн `getCurrentLongWave()` функц нь:

1. Тухайн огнооны хамгийн сүүлийн "ээлж" (wave)-ийг олно.
2. Хэрэв тэр ээлжинд 3 хүн дүүрсэн бөгөөд сүүлийн бүртгэлээс хойш 72 цаг өнгөрөөгүй бол — **"дүүрсэн"** гэж буцаана, шинэ бүртгэл хийхийг хориглоно.
3. Хэрэв 72 цаг өнгөрсөн бол — автоматаар **шинэ ээлж нээгдэнэ** (тусгай cron job хэрэггүй, хүсэлт бүр дээр тооцоолж байна).

Ингэснээр таны тайлбарласан логик яг таарч байна: 3 хүн бүртгүүлмэгц тухайн огноо 72 цагийн турш идэвхгүй болж, хугацаа дуусмагц дараагийн иргэд бүртгүүлэх боломжтой болно.

## Зураг нэмэх

Танилцуулга, мэдээ мэдээлэлд зураг нэмэхийн тулд:
1. Зургаа `frontend/assets/` фолдерт хуулна (жишээ: `frontend/assets/mendee1.jpg`).
2. Admin самбарын "Мэдээ нэмэх" хэсэгт зургийн URL талбарт `/assets/mendee1.jpg` гэж бичнэ.

Мөн `frontend/assets/emblem.svg`-ийг байгууллагын жинхэнэ албан ёсны сүлд/лого зургаар сольж болно (`.svg`, `.png` эсвэл `.jpg` — `index.html` доторх `<img src="assets/emblem.svg">` мөрийг тохируулна).

## API товч жагсаалт

| Method | Endpoint | Тайлбар |
|---|---|---|
| POST | `/api/auth/register-or-login` | Иргэн бүртгүүлэх/нэвтрэх |
| GET | `/api/queue/long/:date` | Тухайн өдрийн урт хугцааны эргэлтийн байдал |
| POST | `/api/queue/long/:date` | Урт хугцааны эргэлтэд бүртгүүлэх |
| GET | `/api/queue/simple/:date` | Энгийн эргэлтийн жагсаалт |
| POST | `/api/queue/simple/:date` | Энгийн эргэлтэд бүртгүүлэх |
| GET | `/api/queue/long-status/:YYYY-MM` | Тухайн сарын дүүрсэн огноонууд (календарьт зориулав) |
| POST | `/api/feedback` | Санал хүсэлт илгээх |
| GET | `/api/feedback/mine` | Миний илгээсэн хүсэлтүүд |
| GET | `/api/news` | Мэдээний жагсаалт (нээлттэй) |
| POST | `/api/admin/login` | Admin нэвтрэх |
| GET | `/api/admin/users` \| `/queue/long` \| `/queue/simple` \| `/feedback` | Admin харах (токен шаардана) |
| POST | `/api/admin/news` | Мэдээ нийтлэх (admin) |
| PATCH | `/api/admin/feedback/:id/status` | Хүсэлтийн статус солих (admin) |

Бүх `/api/queue/*` болон `/api/feedback/*` route-ууд иргэний нэвтрэлтийн токен (`Authorization: Bearer <token>`) шаарддаг.
